# Описание работы метеостанции — шаг за шагом

## Аппаратный состав

| Компонент     | Роль                                              |
|---------------|---------------------------------------------------|
| Raspberry Pi 4 | Центральный вычислительный модуль                |
| BME280        | Датчик температуры, влажности и давления (I²C)   |
| GPS-модуль    | Координаты и высота для API-прогноза (UART)      |
| TFT-дисплей   | Визуализация данных и прогноза (SPI)             |

---

## Запуск системы (systemd → main.py)

При включении Raspberry Pi операционная система автоматически запускает
службу `weather-station` через systemd. Служба выполняет `main.py`.

**Инициализация (однократно при старте):**

1. Загружается конфигурация (`config.py`): пути к файлам, параметры LSTM,
   интервалы работы.
2. Создаётся `DataStore` — обёртка над `weather_history.db`, хранящая
   исторические показания сенсора.
3. Загружается `LSTMForecaster`:
   - Считываются веса нейросети из `lstm_weights.npz` (ERA5, заморожены).
   - Считывается нормировочный scaler (`lstm_scaler.json`).
   - Загружается `CorrectionModel` из `correction_weights.npz`.
4. Создаётся `RuleForecaster` — правиловая резервная система.
5. Создаётся `HybridForecaster`, объединяющий все три движка.
6. Создаётся `HazardDetector` — детектор предвестников опасных явлений.
7. Создаётся `ResearchCollector` — запись данных в `research_data.db`.
8. Инициализируется TFT-дисплей.

## Основной цикл (каждые ~1 секунду)

Каждую секунду `main.py` выполняет следующую последовательность:

### Шаг 1 — Опрос сенсоров

**BME280** (I²C, ~18 мс):
- Считываются температура (°C), влажность (%), давление (hPa).
- Результат: объект `WeatherData` с меткой времени UTC.

**GPS** (UART, ~17 мс в норме, до 11 с при потере сигнала):
- Считывается NMEA-поток, парсятся координаты и высота.
- Результат: объект `GpsData` (широта, долгота, высота, fix_quality).

### Шаг 2 — Сохранение в историческую базу

Показание BME280 записывается в `weather_history.db` (таблица `readings`).
База используется LSTM как источник обучающих и инференс-данных.

### Шаг 3 — Прогноз (раз в N секунд, настраивается в config.py)

`HybridForecaster.predict()` перебирает движки по приоритету:

```
┌─────────────────────────────────────────────────────────────┐
│  1. Online API  ← интернет доступен И GPS имеет fix?        │
│     • Запрос к Open-Meteo по координатам GPS                │
│     • Возвращает temp/pressure на +1h/+2h/+3h               │
│     • Если запрос упал — переход к п.2                      │
├─────────────────────────────────────────────────────────────┤
│  2. LSTM + Коррекция  ← достаточно локальных данных?        │
│     • Ресемплирование истории в 5-минутные бакеты           │
│     • Нормализация → инференс замороженной LSTM             │
│     • Денормализация → сырой прогноз (temp, pressure ×3)   │
│     • CorrectionModel: добавляет поправку Δtemp, Δpressure  │
│     • Если LSTM не готова — переход к п.3                   │
├─────────────────────────────────────────────────────────────┤
│  3. Правиловой (rule-based)  ← всегда доступен              │
│     • Линейная экстраполяция тренда давления                │
│     • Эмпирические пороги влажности и скорости падения P   │
└─────────────────────────────────────────────────────────────┘
```

Результат: объект `HybridForecastResult` — прогноз + метаданные о режиме.

### Детали LSTM-инференса (шаг 3, пункт 2)

1. Из `DataStore` берутся последние `SEQUENCE_LENGTH` (=12) пятиминутных
   бакетов — окно в 1 час.
2. Данные нормируются в [0, 1] по min/max scaler'у ERA5-обучения.
3. Тензор формы `(1, 12, 3)` подаётся в двухслойную LSTM
   (LSTM(64) → Dropout → LSTM(64) → Dense(9)).
4. Выход: 9 значений — [temp, hum, pressure] для каждого из трёх горизонтов
   (+1ч, +2ч, +3ч в шагах по 5 мин: шаг 12, 24, 36).
5. Денормализация → сырой прогноз базовой LSTM.

### Детали коррекционной модели

Поверх сырого прогноза LSTM применяется `CorrectionModel`:

**Входы (12 признаков):**
- Предсказанные LSTM temp_1h, temp_2h, temp_3h
- Предсказанные LSTM pressure_1h, pressure_2h, pressure_3h
- Тренд давления (pressure_3h − текущее давление)
- sin/cos часа суток (циклические признаки)
- sin/cos дня недели
- **Текущая температура сенсора** — ключевой признак для определения
  климатического режима (жара/норма/холод)

**Архитектура:** Dense(64, ReLU) → Dropout(0.1) → Dense(32, ReLU) → Dense(6)

**Выход:** 6 поправок [Δtemp_1h, Δtemp_2h, Δtemp_3h, Δpres_1h, Δpres_2h, Δpres_3h],
ограниченных в диапазоне ±8°C / ±10 hPa.

Скорректированный прогноз: `temp_corrected = temp_lstm + Δtemp`.

### Шаг 4 — Детектор опасных явлений

`HazardDetector.detect()` анализирует последние 60 показаний (~1 час):
- Линейная регрессия давления за 1 час и за 10 минут (hPa/ч).
- Падение температуры за 30 минут.
- Проверка 7 правил → уровень опасности: NORMAL / WATCH / WARNING / DANGER.

Примеры:
- `P(1ч) < −6 hPa и влажность > 80%` → DANGER "Гроза/Шквал"
- `P(10мин) < −1.5 hPa и ΔT > 3°C` → DANGER "Холодный фронт"
- `P(1ч) < −3 hPa и влажность > 70%` → WARNING "Гроза вероятна"

### Шаг 5 — Вывод

**Терминал** (`terminal_display.py`):
- Текущие показания сенсора, GPS-координаты.
- Предупреждение об опасных явлениях (если уровень > NORMAL).
- Таблица прогноза: +1ч/+2ч/+3ч, режим (Online/LSTM/Rule-based).

**TFT-дисплей** (`tft_display.py`):
- Графическое отображение температуры, влажности, давления.
- Значок режима прогноза и цветная полоса опасности.

### Шаг 6 — Запись исследовательских данных

`ResearchCollector` пишет в `research_data.db`:
- **sensor_log**: текущие показания BME280 и GPS.
- **forecast_log**: прогноз с меткой режима (online / lstm / lstm_corrected /
  rule-based).
- **forecast_verification** (через 1/2/3 часа): автоматическое сравнение
  прогноза с тем, что реально показал сенсор. Заполняются поля
  `signed_error_temp_Nh` и `signed_error_pres_Nh`.
- **hazard_log**: запись при смене уровня опасности.

Верификация работает так: каждый цикл система ищет в `forecast_log`
прогнозы, сделанные ровно 1, 2 и 3 часа назад, и сравнивает их с
текущим показанием сенсора.

## Обучение коррекционной модели (ручное, раз в 1–2 недели)

```
python research/calibration_cli.py train-correction
```

1. Из `research_data.db` выбираются верифицированные прогнозы LSTM
   за последние 7 дней (скользящее окно).
2. К каждой строке подтягивается реальная температура сенсора
   из `weather_history.db` на момент прогноза (ATTACH + подзапрос).
3. Формируется матрица признаков X (N×12) и матрица ошибок y (N×6).
4. Обучается Dense-сеть: 80% данных — обучение, 20% — валидация,
   EarlyStopping по val_loss.
5. Новые веса сохраняются в `correction_weights.npz` и сразу загружаются
   в работающую систему при следующем старте.

## Валидация (для диплома)

```
python research/calibration_cli.py validate
```

1. Загружаются все записи из `weather_history.db`.
2. Временной срез 70/30: первые 70% — обучающая выборка, последние 30% —
   отложенная (held-out), эмулирующая незнакомые данные.
3. На отложенной выборке запускается LSTM-инференс (батч).
4. К предсказаниям применяется текущая коррекционная модель.
5. Из `research_data.db` подтягивается MAE Online API за тот же период.
6. Печатается сравнительная таблица: Базовый LSTM / LSTM+Коррекция / Online API.
